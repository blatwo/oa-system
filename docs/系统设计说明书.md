## 1. 系统架构

```
浏览器 (React SPA, Vite 构建)
    │ fetch() JSON over HTTP
    ▼
FastAPI Server (Python, Uvicorn)    ← 静态文件托管 (dist/)
    │ sqlite3 + urllib
    ▼                          ▼
oa_data.db (SQLite)          OpenAI 兼容 API
                                   (DeepSeek 等)
```

启动时 `init_db()` 自动：备份现有 DB → 建表/加列（不删库）→ 种子数据仅新表。

## 5. 关键功能设计

### 5.1 项目状态达成条件（V6.0 重点）

**业务问题**：项目状态可以直接改成"已完成"，缺乏标准化质量门。

**实现方案**：
1. **全局模板**（`project_id=0`）：维护"已完成"状态需要满足的标准条件
2. **项目级条件**：每个项目从模板复制，可独立调整
3. **状态校验**：状态变更为"已完成"时，前端预检 + 后端强制校验（HTTP 400）
4. **审计日志**：所有变更记录到 `project_status_condition_logs`

**数据库迁移**：
```sql
CREATE TABLE project_status_conditions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL,  -- 0 表示全局模板
  target_status TEXT NOT NULL,
  condition_text TEXT NOT NULL,
  is_met INTEGER DEFAULT 0,
  met_date TEXT, met_by TEXT,
  sort_order INTEGER DEFAULT 0,
  created_at TEXT, updated_at TEXT
);
CREATE TABLE project_status_condition_logs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL,
  condition_id INTEGER,
  action TEXT,  -- create/update/delete/met/unmet
  detail TEXT,
  operator TEXT,
  created_at TEXT
);
```

### 5.2 AI 内容质疑（V6.0 重点）

**业务问题**：工程师录入工作记录时过于简略，缺少关键细节。

**实现方案**：
- 调用 LLM 分析 `content` 字段
- 检查项：
  - 提到人名但缺少单位/职位
  - 提到数据库可编程对象（存储过程、函数、触发器、视图等）但缺少数量
  - 问题/故障描述过于笼统
  - 缺少关键操作细节、结果/产出
  - 逻辑连贯性
- 返回结构化 JSON `{questions: [{issue, suggestion, severity}]}`
- 前端以 Dialog 展示，无问题显示绿色"完整度良好"

### 5.3 实用工具集（V6.0 重点）

| 工具 | 用途 |
|------|------|
| SysidAnalyzer | 解析 PostgreSQL system identifier（高32位时间戳/中20位微秒/低12位PID） |
| LsnAnalyzer | 解析 PostgreSQL LSN（XLogRecPtr 八字节结构） |
| PatroniEtcdDebug | Patroni 故障排查命令手册 + etcd 一致性检查 |

## 6. 数据保护机制 🛡️

init_db() 启动流程：
1. **备份**：DB 存在 → 复制为 `.bak.YYYYMMDD_HHMMSS`，保留最近 3 份
2. **建表**：`CREATE TABLE IF NOT EXISTS`（已有表不动）
3. **加列**：`ALTER TABLE ADD COLUMN` try/except（已有列不动）
4. **种子**：`INSERT OR IGNORE` 仅对新表生效
